## 文档说明

### 1.1 目的

本文档详细定义`知阁·舟坊`系统中账号、会话、权限相关的37个业务场景，包括会话生命周期管理、强制下线机制、账号封禁与注销、多设备会话控制、异地登录风控等核心功能的需求规格。

### 1.2 适用范围

- 前端页面：登录页、个人工作台、管理员后台、WorkspaceHub、组件库页面等系统全部页面
- 后端API：Auth API、User API、Admin API等
- 中间件：AuthCheck、ActivityMonitor等
- 数据库：User表、Session表、UserDevice表等

### 1.3 设计原则

1. **安全性优先**：任何操作优先保证账号安全
2. **用户体验**：异常情况需给出明确提示，避免用户困惑
3. **可追溯**：所有状态变更需记录日志
4. **容错性**：异常情况不能导致系统崩溃

## 术语定义

| 术语 | 定义 |
|------|------|
| AccessToken | 短期访问令牌，用于API鉴权，默认有效期2小时 |
| RefreshToken | 刷新令牌，用于续期AccessToken，默认有效期7天 |
| Session | 用户会话，包含sessionToken和会话相关信息 |
| Cookie | 浏览器存储会话信息的载体 |
| 强制下线 | 管理员或系统主动使某会话失效 |
| 挤线 | 新登录使旧会话失效 |
| 冷静期 | 用户申请注销后等待正式删除的时间，默认7天 |
| 风控 | 风险控制，用于检测异常登录行为 |
| SSO | 单点登录，支持飞书、钉钉等第三方登录 |
| 单设备限制 | 同一账号同时只能有一个设备在线 |
| 空间踢出 | 将用户从某个工作空间移除，但不踢出系统 |

### 第一类：会话超时与生命周期管理

---

#### 场景1：普通用户长时间无操作

**场景ID**：SCENARIO_001
**场景名称**：全部用户长时间无操作自动重定向登录页
**优先级**：P0（核心功能）
**实现状态**：待实现

##### 4.1.1 前置条件

1. 用户已成功登录系统
2. 存在有效的sessionToken
3. 用户在操作工坊或工作台页面

##### 4.1.2 触发条件

- 用户在5分钟（300秒）内无任何操作
- 操作包括：鼠标移动、鼠标点击、键盘输入、滚动页面

##### 4.1.3 详细流程

```
1. [前端] ActivityMonitor组件在用户首次进入页面时启动监测
2. [前端] 每当用户有操作时，更新lastActivityTime时间戳
3. [前端] 每60秒检查一次：当前时间 - lastActivityTime > 5分钟？
   - 是：执行步骤4
   - 否：继续监测
4. [前端] 调用 /api/auth/touch 接口，携带当前sessionToken
5. [后端] 验证sessionToken有效性
   - 有效：返回 { valid: true, expiresIn: seconds }
   - 无效/过期：返回对应错误码
6. [前端] 收到SESSION_TIMEOUT错误码
7. [前端] 显示Toast提示："您已长时间未操作，请重新登录"
8. [前端] 延迟1.5秒后，跳转到登录页，URL参数：/auth/login?reason=SESSION_TIMEOUT
9. [登录页] 解析reason参数，显示对应提示信息
```

##### 4.1.4 UI交互

| 步骤 | 用户操作 | 系统响应 |
|------|----------|----------|
| 1 | 无操作5分钟 | 系统无响应，继续等待 |
| 2 | 5分钟到达 | 显示Toast："您已长时间未操作，请重新登录" |
| 3 | Toast消失（3秒后） | 自动跳转登录页 |
| 4 | 到达登录页 | 显示："因长时间未操作，您已自动退出登录" |

##### 4.1.5 接口规范

**请求**：`POST /api/auth/touch`
```json
{
  "sessionToken": "xxx"
}
```

**响应（成功）**：
```json
{
  "valid": true,
  "expiresIn": 1800,
  "serverTime": "2026-05-12T10:00:00Z"
}
```

**响应（失败）**：
```json
{
  "error": "SESSION_TIMEOUT",
  "message": "您已长时间未操作，请重新登录"
}
```

##### 4.1.6 代码位置

| 组件 | 文件路径 | 职责 |
|------|----------|------|
| ActivityMonitor | `src/components/ActivityMonitor.tsx` | 监测用户活动，调用touch接口 |
| AuthCheck | `src/components/AuthCheck.tsx` | 拦截401响应，处理错误码 |
| LoginPage | `src/app/auth/login/page.tsx` | 解析reason参数，显示提示 |

##### 4.1.7 边界情况

1. **用户在多个标签页操作**：每个标签页独立计时，但任一标签页操作会同步更新服务端lastActiveAt
2. **用户切换到其他标签页**：仍然计时，因为tab可能只是最小化但页面仍在运行
3. **网络中断时超时**：恢复网络后，touch接口会返回错误，前端正确处理跳转
4. **用户电脑休眠**：休眠超过5分钟，恢复后会被踢出登录

#### 场景3：根据"记住我"决定生命周期

**场景ID**：SCENARIO_003
**场景名称**：根据"记住我"选项决定会话生命周期
**优先级**：P0（核心功能）
**实现状态**：待实现

##### 4.3.1 生命周期规则

| 选项 | AccessToken有效期 | RefreshToken有效期 | Session有效期 |
|------|------------------|-------------------|---------------|
| 不勾选（默认） | 2小时 | 7天 | 2小时（活动） |
| 勾选"记住我" | 24小时 | 30天 | 24小时（活动） |

##### 4.3.2 详细流程（登录）

```
1. [登录页] 用户输入账号密码
2. [登录页] 用户勾选/不勾选"记住我"选项
3. [登录页] 提交登录请求到 /api/auth/login
4. [后端] 验证账号密码
5. [后端] 根据rememberMe参数设置不同过期时间
   - rememberMe=true: sessionExpiresAt = now + 30天
   - rememberMe=false: sessionExpiresAt = now + 2小时
6. [后端] 创建Session记录
7. [后端] 生成AuthToken，设置HttpOnly Cookie
8. [后端] 返回登录成功，包含用户信息和token
9. [前端] 存储token到localStorage
10. [前端] 根据重定向目标路径（redirectPath）跳转到来源页面或工作台：
    - 普通用户若尝试访问管理员页面（如 /admin 路由），将被重定向至工作空间枢纽 `/workspace-hub`；
    - 管理员默认登录或访问普通页面时，将前往目标普通页面或默认的工作空间枢纽 `/workspace-hub`，不再强制重定向至管理员后台 `/admin`；
    - 正常访问或有特定深度链接时，跳转至原目标页面。
```

##### 4.3.3 详细流程（会话验证）

```
1. [前端] 发起API请求
2. [后端] 验证Cookie中的sessionToken
3. [后端] 检查sessionExpiresAt是否过期
   - 未过期：允许请求
   - 已过期：检查是否为rememberMe用户
     - 是：检查RefreshToken是否有效
       - 有效：自动续期，返回新token
       - 无效：返回SESSION_EXPIRED
     - 否：返回SESSION_EXPIRED
4. [前端] 收到SESSION_EXPIRED，跳转登录页
```

##### 4.3.4 UI规范

- **复选框文案**："7天内免登录"（勾选后显示）
- **位置**：登录按钮下方
- **样式**：使用系统Checkbox组件
- **默认值**：不勾选

##### 4.3.5 代码位置

| 组件 | 文件路径 | 职责 |
|------|----------|------|
| LoginPage | `src/app/auth/login/page.tsx` | 收集rememberMe参数 |
| Auth API | `src/app/api/auth/login/route.ts` | 根据参数设置不同过期时间 |
| Session验证 | `src/middleware/auth.ts` | 验证session有效性 |

async function checkSession(req) {
  const session = await getSession(req);
  if (!session) return null;

  const user = await prisma.user.findUnique({
    where: { id: session.userId },
    select: { role: true, passwordChangedAt: true }

  if (user.role !== session.cachedRole) {

#### 场景19：后端重启后会话失效

**场景ID**：SCENARIO_019

**优先级**：P0（核心功能）
**实现状态**：待实现

##### 4.19.1 原因分析

- 服务器重启后，内存中的Session数据丢失
- 如果Session存储在内存中，会导致所有会话失效
- 如果Session存储在Redis中，则不受影响

##### 4.19.2 推荐方案

```
使用JWT Token替代Session：
- JWT自包含用户信息
- 不依赖服务端存储
- 服务重启不影响有效性
- 结合RefreshToken实现续期
```

##### 4.19.3 JWT方案详细流程

```
1. [用户] 登录成功
2. [后端] 生成JWT AccessToken（2小时）
3. [后端] 生成RefreshToken（7天）
4. [后端] RefreshToken存储到Redis（持久化）
5. [后端] 重启，内存清空
6. [用户] 发起API请求
7. [后端] 验证JWT（自包含，无需内存）
8. [后端] 允许请求
9. [用户] AccessToken快过期（<10分钟）
10. [前端] 使用RefreshToken获取新AccessToken
11. [后端] 验证RefreshToken（Redis查询）
12. [后端] 生成新AccessToken
```

// useTokenRefresh hook
function useTokenRefresh() {
  const refreshInterval = 50 * 60 * 1000; // 50分钟

  useEffect(() => {
    const timer = setInterval(async () => {
      try {
        const res = await fetch('/api/auth/refresh', {
          method: 'POST',
          credentials: 'include'
        });
        if (res.ok) {
          const data = await res.json();
          localStorage.setItem('token', data.accessToken);
        }
      } catch (e) {
        console.error('Token refresh failed');
      }
    }, refreshInterval);

    return () => clearInterval(timer);
  }, []);
}

// 全局API请求拦截
async function request(url, options) {
  const res = await fetch(url, {
    ...options,
    credentials: 'include'  // 确保发送Cookie
  });

  if (res.status === 401) {
    // 尝试刷新Token
    const refreshRes = await fetch('/api/auth/refresh', {
      method: 'POST',
      credentials: 'include'
    });

    if (refreshRes.ok) {
      // 重新发起原请求
      return fetch(url, options);
    }

    // 刷新失败，跳转登录
    window.location.href = '/auth/login?reason=TOKEN_INVALID';
  }

  return res;
}

4. [前端] 用户在工坊页面成功绑定组件后，返回空间首页，操作工坊侧边栏在初始化时向 `/api/studio?action=bound&workspaceId=...` 发送异步请求拉取最新绑定的组件 ID 列表，并与系统默认的可用组件进行去重合并，最终动态呈现在 10 大软件工程阶段中，形成完美的业务功能闭环。

#### 场景44：空间配额自动激活与事务级初始化

**场景ID**：SCENARIO_044
**场景名称**：个人/企业空间创建时自动在事务中初始化配额记录并绑定初始Token余额
**优先级**：P0（核心功能）
**实现状态**：已实现

##### 1. 前置条件
1. 用户已成功登录系统。
2. 触发个人空间创建或企业空间开通的 POST 请求。

##### 2. 触发条件
- 用户在工作台（`/workspace-hub`）点击创建自定义个人空间，或者提交申请开通企业空间。

##### 3. 详细流程
1. [后端] 接收到空间创建请求（`/api/workspace/create` 或 `/api/workspace/create-enterprise`）。
2. [后端] 根据用户的当前会员等级（`membershipLevel`，如 `FREE`, `GOLD`, `DIAMOND`等）或申请的企业空间版本（`STANDARD`, `PRO`, `ENTERPRISE`）计算对应的初始 Token 配额（如个人普通用户免费版 10,000 点，企业专业版 100,000 点，企业至尊版 500,000 点等）。
3. [后端] 启动数据库事务。
4. [后端] 在同一事务中先创建 `workspace` 记录本身。
5. [后端] 嵌套创建工作空间成员 `workspacemember` 记录，角色设为 `OWNER`。
6. [后端] 嵌套创建配额 `workspacequota` 记录，绑定当前工作空间 ID（`workspaceId`），并赋予计算出的初始 Token 余额 `tokenBalance`，写入对应的会员等级 ID（`membershipLevelId`）。
7. [后端] 提交事务。如果任何一步失败，自动回滚以确保数据一致性，避免空间被创建却无配额导致的余额溢出或空指针错误。
8. [后端] 成功后返回空间实体，前端同步更新本地工作空间列表。

##### 4. 接口规范
**普通空间创建请求**：`POST /api/workspace/create`
**企业空间创建请求**：`POST /api/workspace/create-enterprise`
**响应（成功）**：
```json
{
  "success": true,
  "workspace": {
    "id": "ws-xxx",
    "name": "企业协作工作空间",
    "type": "ENTERPRISE",
    "workspacequota": {
      "id": "quota-uuid",
      "workspaceId": "ws-xxx",
      "membershipLevelId": "FREE",
      "tokenBalance": "100000"
    }
  }
}
```

##### 5. 代码位置
- 个人空间创建：`src/app/api/workspace/create/route.ts`
- 企业空间创建：`src/app/api/workspace/create-enterprise/route.ts`

#### 场景45：企业空间岗位细粒度组件权限强拦截

**场景ID**：SCENARIO_045
**场景名称**：根据企业空间成员的岗位与组件权限策略进行全链路强拦截与置灰引导
**优先级**：P0（核心功能）
**实现状态**：已实现

##### 1. 前置条件
1. 用户已登录并处于某个企业工作空间（`Workspace`）中。
2. 企业空间下定义了多种岗位（`WorkspacePost`），并通过岗位成员关联表（`PostMember`）将当前用户绑定至特定岗位。
3. 岗位与效能组件之间通过 `ComponentPermission` 关系表配置了权限规则。

##### 2. 触发条件
- 企业成员访问效能组件工坊页面（`/studio`），或者通过 API 调用某组件的模拟运行（`simulate`）、空间绑定（`bind`）、或者实际调用（`use`）。

##### 3. 详细流程
1. [后端] 任何涉及组件交互的 API（`/api/studio`）接收到请求时，自动校验当前传入的工作空间 ID 以及当前用户的岗位。
2. [后端] 系统通过 `getRestrictedComponentIds(workspaceId, userId)` 算法进行岗位权限汇总：
   - 若当前工作空间为个人空间（`PERSONAL`），则无岗位受限策略，默认全部组件有权使用。
   - 若为企业空间，系统查询该用户在该企业空间下绑定的所有岗位（`postId`）。
   - 若用户在企业中未被分派任何岗位，系统触发默认安全策略，默认受限使用高敏感/高算力消耗组件（如 C01, C02, C03）。
   - 若被分派了岗位，系统拉取这些岗位对应的 `ComponentPermission` 权限记录。凡是未明确赋予 `canExecute` 权限的组件，均归入受限组件列表（`restrictedComponentIds`）。
3. [后端] 在执行 `action === "simulate"`、`action === "bind"`、`action === "use"`时，若请求的组件 ID 存在于受限组件列表中，则立即抛出 403 Forbidden 错误并终止操作：`{ error: "岗位权限受限，无法使用此组件" }`。
4. [前端] 效能组件工坊（`/studio`）加载时，首选调用后端接口拉取当前空间的 `restrictedComponentIds` 受限组件 ID 数组。
5. [前端] 联动渲染置灰逻辑：
   - 遍历组件列表（列表视图、网格卡片、快捷访问区、组件详情弹窗等）。
   - 若组件 ID 被判定受限，其交互按钮（“模拟运行”、“绑定至工作空间”、“立即使用”）在视觉上变更为 Slate 灰色置灰状态，按钮文案修改为统一的“岗位权限受限”。
   - 在置灰按钮的点击事件（`handleSimulateRun`, `handleBindToWorkspace`, `useComponent`）首行加入防御性物理拦截，点击后静默拦截或弹出 Toast 友情警示，达成前后台双重安全闭环。

##### 4. 接口规范
**获取受限组件列表**：`GET /api/studio?action=restricted&workspaceId=ws-xxx`
**响应（成功）**：
```json
{
  "restrictedComponentIds": ["C01", "C02"]
}
```

##### 5. 代码位置
- 权限拦截 API：`src/app/api/studio/route.ts`
- 前端置灰与拦截：`src/app/studio/page.tsx`

#### 场景47：能力宣发页面体验闭环与营销导流

**场景ID**：SCENARIO_047
**场景名称**：产品能力展示页针对未登录及已登录用户精细化营销导流与工坊闭环交互
**优先级**：P0（核心功能）
**实现状态**：已实现

##### 1. 前置条件
1. 用户浏览系统的产品能力宣发与解决方案页面（`/capabilities`）。

##### 2. 触发条件
- 用户点击各技术解决方案卡片下方的“体验”或“立即运行”按钮。

##### 3. 详细流程
1. [前端] 页面检测当前的登录状态：
   - **未登录/游客状态**：
     - 用户点击“体验”或“立即运行”按钮，系统自动弹窗精美且符合设计规范的“申请演示与试用弹窗”（`DemoModal`）。
     - 弹窗中收集用户的姓名、公司、联系方式以及意向说明。
     - 用户提交表单后，调用 `/api/demo-request` 后端接口将申请入库，并弹出“申请成功，我们的专员将在24小时内与您联系”的确认 Toast，形成营销与获取商机线索的业务闭环。
   - **已登录状态**：
     - 去除原有的本页冗余运行逻辑（此前需在本页重复选择空间并模拟运行，操作极其割裂且功能重复）。
     - 侧边栏及卡片区域自动变为醒目的绿色高亮登录导流卡，直接展示用户的“当前活跃工作空间”。
     - “立即运行”按钮一律升级重构为绿色醒目的“直接在工坊使用”导流按钮。
     - 用户点击该按钮，系统将在路由上自动携带两个核心参数（当前工作空间 `workspaceId` + 当前解决方案所绑定的组件 ID `componentId`），例如：`/studio?workspaceId=ws-xxx&componentId=C01`。
     - 前端执行平滑跳转至效能工坊页面（`/studio`）。
4. [前端] 效能工坊（`/studio`）检测到 URL 中的 `componentId` 参数：
   - 自动开启副作用 Hook 扫描，将列表或卡片视口平滑滚动（`scrollIntoView`）至该目标组件位置。
   - 高亮闪烁该组件卡片，并自动打开该组件的详细运行和配置面板，让用户无缝直接开始运行和体验。这不仅完全去除了产品宣发页与工坊页面的功能冗余，而且完美串联了“认知 -> 导流 -> 落地 -> 生产运行”的全部数据与业务闭环。

##### 5. 代码位置
- 产品能力页：`src/app/capabilities/page.tsx`
- 效能组件工坊：`src/app/studio/page.tsx`
